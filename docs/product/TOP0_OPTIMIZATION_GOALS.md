# WorldBox Writer: 核心优化目标与落地计划

本文档定义了系统从当前版本（基础流程已跑通）向高可用、高质量创作工具演进的具体技术目标与执行路径。

## 1. 核心优化维度

### 1.1 交互体验 (UX)
*   **多世界线分支管理**
    *   概念：允许用户在任意历史节点（StoryNode）创建分支，独立推演不同的剧情走向。
    *   实现：引入树状或图状的数据结构来管理 Session 的历史记录，支持节点状态的切换（Checkout）和合并（Merge）。前端增加分支管理视图。
*   **动态设定集 (World Wiki)**
    *   概念：支持用户在前端界面直接增删改查角色、地点和物品设定。
    *   实现：提供基于卡片或列表的编辑界面，数据变动实时同步至后端数据库和 ChromaDB 向量库。
*   **实时推理过程展示**
    *   概念：在 Agent 推理过程中，将中间状态和思考过程实时展示给用户。
    *   实现：基于 SSE (Server-Sent Events) 或 WebSocket，将 Agent 的内部日志（如 GateKeeper 的判定理由、Director 的大纲调整）流式推送到前端。

### 1.2 系统稳定性 (Stability)
*   **强类型与 Schema 约束**
    *   概念：确保 LLM 输出严格符合预期的 JSON 结构，避免解析错误导致流程中断。
    *   实现：在所有 LLM 调用点强制使用 JSON Schema 校验（如 Pydantic models）。如果解析失败，触发自动重试或降级逻辑。
*   **后端类型安全**
    *   概念：消除 Python 代码中的类型模糊地带。
    *   实现：开启 Mypy `strict=True`，修复所有类型警告，彻底移除 `Any` 类型的滥用。
*   **存储状态一致性**
    *   概念：确保关系型数据库（SQLite）和向量数据库（ChromaDB）的数据始终保持同步。
    *   实现：在写入和删除操作时引入事务机制或补偿逻辑。如果向量库写入失败，必须回滚关系库的变更。

### 1.3 服务可用性 (Availability)
*   **异步任务调度**
    *   概念：将耗时的 LLM 推理和文本生成任务从 API 主线程中剥离。
    *   实现：废弃现有的 `ThreadPoolExecutor`，引入 Celery 或 Temporal 等专业任务队列，支持多 Worker 节点并发处理。
*   **LLM 路由与容错**
    *   概念：在当前 LLM 服务不可用或超时时，自动切换到备用模型。
    *   实现：封装统一的 LLM 客户端层，支持配置主备模型（如主 OpenAI，备 Anthropic 或本地模型），并实现指数退避重试机制。

### 1.5 引擎架构 (Engine Architecture)
*   **真·多智能体独立推演 (True Multi-Agent Topology)**
    *   概念：废弃当前的“上帝视角（God-Mode）”统一 Prompt，为沙盒中的每一个核心角色分配一个物理隔离的 Agent 实例与上下文窗口。
    *   实现：引入“环境服务器（World Engine）”与“角色客户端（Actor Client）”的架构。每个 Actor 只能看到自己所在位置的局部信息（Local Observation），独立做出决策；由 Director 作为 Game Master 负责冲突结算（Conflict Resolution）并推动世界状态更新。
*   **外部 Embedding 模型**
    *   概念：替换当前基于本地哈希的文本检索方式，提高上下文召回的语义准确率。
    *   实现：集成专业的 Embedding 模型（如 OpenAI `text-embedding-3-small` 或本地 `bge-m3`），结合 BM25 实现混合检索（Hybrid Search）。
*   **多模型协同 (MoE 模式)**
    *   概念：根据任务类型（逻辑推演 vs 文本润色）选择最合适的 LLM，平衡成本与效果。
    *   实现：在配置层面解耦 Agent 使用的模型。逻辑推演（Director/GateKeeper）配置低成本/高推理模型，文本生成（Narrator）配置长文本/高文笔模型。

## 2. 阶段性执行计划

### Phase 1: 质量与稳定性 (Quality & Stability)
*优先解决生成质量和系统崩溃问题，确保核心流程可用。*
1.  **强 Schema 约束与自动重试**：实现 LLM 输出校验机制，降低解析失败率。
2.  **多模型协同 (MoE)**：分离推演与渲染模型配置，提升最终文本质量。
3.  **外部 Embedding 与混合检索**：接入真实 Embedding 模型，解决长文本遗忘问题。

### Phase 2: 交互与异步基建 (UX & Async Foundation)
*解耦底层架构，提升前端交互响应速度。*
1.  **异步任务队列**：引入 Celery/Temporal，将推演任务异步化。
2.  **实时推理流推送**：通过 SSE 将 Agent 思考过程推送到前端。
3.  **动态设定集 (CRUD)**：实现前端对设定的增删改查及后端同步。

### Phase 3: 复杂特性与高可用 (Advanced Features & HA)
*实现高级护城河功能，确保生产环境高可用。*
1.  **多世界线分支管理**：重构状态管理，支持历史节点的分支与切换。
2.  **LLM 路由与容错**：实现多 Provider 自动切换与重试机制。
3.  **存储状态一致性**：完善 SQLite 与 ChromaDB 的事务同步机制。

---

## 3. 验收标准 (Metrics)

*   **稳定性**：在包含 50 个节点的长篇推演测试中，因 LLM 输出格式导致的流程中断率 `< 1%`。
*   **性能**：API 接口响应时间（不含 LLM 等待时间）`< 500ms`；异步任务状态查询延迟 `< 100ms`。
*   **准确率**：在混合检索测试集中，历史关键设定（角色关系、重要道具）的召回率 `> 95%`。
*   **并发性**：异步队列支持同时处理至少 5 个独立的生成任务，且互不干扰。
