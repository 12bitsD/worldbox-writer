## 一、产品定位

### 核心矛盾

产品的架构目标是"故事推演引擎"，但用户需要的是"写作助手"。

- 推演引擎的隐喻：Agent 自主推演，人类观察和干预。
- 写作助手的隐喻：人类主导创作，AI 辅助执行。

**当前姿态是"AI 是主角，人类是监工"；应该反过来："人类是作者，AI 是助手"。**

### 一句话定位

> **写长篇小说时，AI 不会忘记前面写了什么。**

比"沙盒式 AI 小说生成系统"更具体、可验证。

### 核心价值

排除花哨功能，只有一个真正不可替代的差异化：

> **在 AI 生成长篇故事时，保持角色和世界的结构性一致性。**

| 底层能力 | 用户感知 |
| :--- | :--- |
| Scene Script 先于渲染 | "这 AI 写的长篇居然没有前后矛盾" |
| 角色隔离推演 | "每个角色都像他自己，不是提线木偶" |
| 分支系统 | "我可以探索不同剧情，选最好的" |
| 分层记忆 | "角色还记得第 3 章发生的事" |

## 三、成熟度评估

**当前状态：L2.5（介于"可管理"和"已定义"之间）**

| 维度 | 当前得分 | 市面平均 | 差距 | 说明 |
| :--- | :---: | :---: | :---: | :--- |
| 功能完整度 | 7/10 | 7/10 | 0 | 核心功能已闭环，缺"最后一公里" |
| **开箱即用** | **3/10** | **7/10** | **-4** | 需本地部署、配置 API、启动前后端 |
| 输出质量 | 6/10 | 7/10 | -1 | 双循环架构理论上更好但待验证 |
| 用户体验 | 5/10 | 7/10 | -2 | 仍偏工程师玩具 |
| **商业化就绪** | **2/10** | **7/10** | **-5** | 无用户体系、无付费、无 SaaS |
| **社区生态** | **1/10** | **6/10** | **-5** | 无模板、社区、教程 |
| 稳定性 | 5/10 | 7/10 | -2 | 缺真实用户压力测试 |

**综合距离：2.4 / 3.0（市面平均线）= 达到 80%**

### 综合判断

> WorldBox Writer 在技术架构和创新性上大幅领先市面产品，但在易用性、商业化和生态建设上严重落后。
> 它是一个优秀的技术原型，但还不是一个可用的产品。
> 核心矛盾是："工程师做了 20+ Sprint 的架构创新，但用户连第一步都迈不出去。"

它像一辆装了 F1 引擎的拖拉机——引擎很强，但车身、轮胎、方向盘都需要换。
好消息是：这些问题都是"换壳"问题，不是"换引擎"问题。

## 四、用户旅程断点

### 阶段 1：首次接触（🔴 高严重度）

- **现状**：clone → 装 Python/Node/pnpm → 创建 `.env` → 配置 LLM API Key → 启动前后端 → 打开浏览器
- **市面标准**：注册 → 输入一句话 → 开始创作（3 分钟内）
- **差距**：30 分钟技术配置 vs 3 分钟开始创作 = **10 倍体验差距**

### 阶段 2：首次产出（🟡 中严重度）

- **现状**：输入一句话 → 等待 Director → 等待 WorldBuilder → 等待推演 → 等待渲染 → 看到第一节点
- **市面标准**：输入一句话 → 10-30 秒看到第一段
- **差距**：首屏等待过长（Sprint 8.5 已优化）

### 阶段 3：持续使用（🟡 中严重度）

- **现状**：核心循环成立（推演 → 干预 → 继续），但缺：
  - 进度保存/恢复的用户感知
  - 作品积累感（无"我的作品库"）
  - 重复使用动力（无成就、无社区）
- **差距**：缺"作品管理"心智模型

### 阶段 4：产出作品（🟢 低严重度）

- **现状**：支持 TXT/MD/HTML/DOCX/PDF 导出
- **差距**：
  - 缺"章节化"组织（当前是"节点"）
  - 导出前的编辑器体验有限

## 六、交互原则

当前 UI 是三栏仪表盘，像 Grafana 或 Kubernetes Dashboard——
但用户是**写小说的人**，他们用的工具是 Word / Scrivener / Notion。

### 目标

**一个写作工具，不是一个推演仪表盘。**

### 具体原则

1. **主区域是文字，不是数据**。用户 90% 时间在读/写。
2. **干预是选择题，不是填空题**。提供 2-3 个 AI 生成的选项 + "自己写"入口。
3. **推演过程对用户透明但不打扰**。只显示"正在写下一章..."，不暴露"Director 正在生成 Scene Plan"。
4. **设定集是参考，不是主界面**。折叠在侧栏或抽屉。
5. **编辑是第一公民**。用户能直接改 AI 文字，AI 基于修改后的版本继续。

### 干预机制重设计

| 当前 | 目标 |
| :--- | :--- |
| 每节点暂停 → 用户输入文字 → 继续 | 默认不暂停，AI 自动推演 3-5 章 |
| 太频繁、太重、用户不知写什么 | 剧情关键分歧点暂停，提供选择题 |
| | 用户可主动暂停、回退到任意章节 |

## 七、应砍掉/隐藏的功能

这些功能属于"工程师玩具"，应从用户主界面移除或隐藏到高阶入口：

- **Telemetry 面板**：没有用户想看 LLM 调用日志
- **Prompt Inspector**：开发调试工具，不是产品功能
- **关系图谱可视化**：看起来酷但角色少时列表就够
- **双循环对比报告**：纯工程产物
- **Diagnostics API**：开发者工具
- **多模型路由/fallback**：用户不关心底层用什么模型
- **"推演日志"面板**：把 Agent 内部状态暴露给用户，违反产品原则
- **T1/T2/T3 推演深度选择**：用户不理解"tick"，应叫"章节数"或不暴露
- **"创世神"隐喻**：让产品显得像游戏而非严肃写作工具

## 八、差异化策略

### 应走"写作工具"路线，不是"互动叙事"路线

1. **互动叙事市场小**：AI Dungeon 已证明天花板低、留存差
2. **写作工具市场大**：Sudowrite 年收入千万美金
3. **核心技术优势在写作场景更有价值**："长篇一致性"对写小说是刚需

### 与竞品的定位差异

| 竞品 | 定位 | WorldBox Writer 差异定位 |
| :--- | :--- | :--- |
| ChatGPT | 通用对话 | "专门写长篇，不会写到后面忘了前面" |
| Sudowrite | 章节级助手 | "能帮你规划整本书结构" |
| NovelAI | 模型驱动自由写作 | "你能控制故事走向" |
| Novelcrafter | 大纲+章节管理 | "AI 真的能帮你写，角色不会崩" |

## 九、产品路线图

### Phase 0：能用（2 周） — 当前阶段

- 修好 UI 交互 bug
- 修好输出质量（重写 Narrator prompt、换模型）
- 修好 initializing 卡住问题

### Phase 1：好用（1 个月）

- 重新设计主界面（富文本编辑器为主）
- 选择题式干预（替代文字输入）
- 角色名用 LLM 生成真实人名
- 章节化组织（替代 tick/node）
- 一键导出 DOCX

### Phase 2：想用（2 个月）

- 用户编辑 AI 文本后继续推演
- 大纲导入
- 模板市场（武侠/科幻/言情等预设）
- 进度保存/恢复的明确 UI
- 风格预设（网文/文学/轻小说）

### Phase 3：离不开（3-12 个月）

- SaaS 部署（Docker Compose 或 HuggingFace Spaces 一键部署）
- 长篇完整章节管理
- 角色弧线追踪
- 社区分享 / 模板生态
- 移动端响应式

## 十、最终判断

WorldBox Writer 的**技术架构是好产品**，但**当前实现不是好产品**。

核心差距不在功能，而在：

1. **无法触达**（需本地部署）
2. **无法理解**（缺引导和模板）
3. **无法持续**（缺作品管理和留存机制）

好消息：这些差距都是"最后一公里"问题，不是架构问题。
技术地基已经打好，接下来需要的是**产品化投入**，而不是技术投入。

**距离好产品的距离：约 2-3 个月的集中产品化工作。**

**最该做的一件事**：把 Narrator 输出质量拉到"可读的小说"水平。
一旦用户看到 AI 写出了像样的第一章，其他所有问题都有了解决的动力。
