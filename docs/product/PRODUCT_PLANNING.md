# WorldBox Writer: 产品演进规划 (Product Planning)

**文档状态**：Active (v0.5.0+)
**作者**：Manus AI

本文档基于 WorldBox Writer 的产品愿景，自顶向下（Top-Down）对未来版本进行规划。结构为：**Product Vision → Release Goals (阶段大目标) → Epics (史诗故事) → User Stories (用户故事) → Sprints (冲刺排期)**。

---

## 1. 产品愿景 (Product Vision)

**"打造一个让创作者以'创世神'视角，通过干预 Agent 沙盒世界来生成长篇小说的生产力工具。"**

WorldBox Writer 的核心差异化在于**从"文本编辑器"升维到"世界引擎"**。我们不追求让 AI 一次性吐出 10 万字，而是让多个 Agent（角色、导演、守卫）在一个有规则的虚拟沙盒中互动，用户通过设定规则和关键节点的干预来引导故事走向，从而解决长篇一致性差、缺乏涌现性的痛点。

---

## 2. 阶段性大目标 (Release Goals)

目前我们已完成 **Release 1 (v0.5.0)**：端到端 MVP，证明了"沙盒推演 + 关键点干预"的技术可行性，并实现了基本的实时流与持久化。

接下来，我们将产品演进划分为三个具有**严格因果递进关系**的核心 Release 阶段。这三个阶段对应了创作者在使用本工具时的认知旅程：**看见世界 (Observe) → 掌控世界 (Control) → 创作作品 (Create)**。

### Release 2: 看见世界 (Visibility) - v0.6.0
**大目标**：让黑盒的推演过程变得直观可见，让创作者能够"看懂"沙盒里正在发生什么。
- **为什么必须先做**：如果用户连角色关系如何变化、Agent 为什么做出某个决定都看不清，就无法做出有意义的干预。
- **核心交付**：动态人物关系图谱、Agent 遥测日志（内部思考过程可见）、事件流增强。

### Release 3: 掌控世界 (Control) - v0.7.0
**大目标**：在"看懂"的基础上，提供更强大的时间线操控能力，让用户能精确引导世界走向。
- **为什么是第二步**：只有在具备了全局视角（Release 2）后，用户才能有针对性地回溯历史节点，尝试不同分支，或调整叙事节奏。
- **核心交付**：时间线回溯与分叉（Undo & Branching）、多分支对比评估、叙事节奏控制器（Pacing Control）。

### Release 4: 创作作品 (Productivity) - v0.8.0 - v1.0.0
**大目标**：从"沙盒玩具"彻底升级为"生产力工具"，支持创作者将推演结果打磨成真正的文学作品。
- **为什么是最后一步**：必须先有足够强大的推演和掌控能力（Release 3），"导入大纲"和"精细润色"才有意义。
- **核心交付**：大纲与预设导入、交互式设定 Wiki、前端富文本编辑器集成、多格式专业导出（PDF/Word）、多模型路由（质量与成本优化）。

## 3. 史诗拆解与用户故事 (Epics & User Stories)

以下重点拆解 Release 2 到 Release 4 的 Epics 和 User Stories。

### Epic 05: 可视化沙盒面板 (Visual Sandbox Dashboard) - *Release 2*
*（继承自 MVP，解决"看见世界"的问题）*

- **US-05.02: 动态人物关系图谱 (P0)**
  - 作为创世神，我希望能查看动态更新的人物关系图谱，以便掌握全局势力变化。
- **US-05.03: 真实 Agent 遥测日志 (P2)**
  - 作为创世神，我希望能看到各 Agent 的内部思考过程（如提议被拒绝的原因），以便了解世界运转的底层逻辑。

### Epic 06: 多世界线与分支管理 (Branching Narratives) - *Release 3*
*（WorldBox Writer 的核心杀手锏，解决"掌控世界"的问题）*

- **US-06.01: 时间线回溯与分叉 (P0)**
  - 作为创世神，我希望能在任意历史节点"存档"并引入新的干预指令，从而衍生出平行世界线，以便探索不同的故事走向（如"如果主角在这里选择背叛会怎样？"）。
- **US-06.02: 多分支对比面板 (P1)**
  - 作为创世神，我希望能同时推演并对比 2-3 条分支大纲，以便选择最满意的一条进行精细渲染。
- **US-06.03: 叙事节奏控制器 (P2)**
  - 作为导演，我希望能设置当前阶段的"基调"（如：平缓日常、紧张战斗），以便 GateKeeper 能据此调整 Agent 提议的通过率。

### Epic 07: 深度世界构建与资产管理 (Deep Worldbuilding) - *Release 4*
*（将设定的控制权交还给人类，解决"创作作品"的问题）*

- **US-07.01: 大纲与预设导入 (P1)**
  - 作为世界构建师，我希望能上传现有的世界观设定或故事大纲 JSON，以便让系统基于我的存稿进行推演，而不是从零生成。
- **US-07.02: 交互式设定集 Wiki (P1)**
  - 作为世界构建师，我希望能在一个类似 Wiki 的界面中增删改查势力、地点和角色设定，修改后自动触发一致性校验。
- **US-07.03: 多格式专业导出 (P1)**
  - 作为创作者，我希望能将推演完成的小说导出为排版精美的 PDF 和 Word 格式，以便直接交付或继续排版。

### Epic 08: 工作流与生产力增强 (Workflow & Productivity) - *Release 4*

- **US-08.01: 前端富文本编辑器集成 (P2)**
  - 作为叙述者，我希望能在 Narrator 渲染完成后，在一个所见即所得的编辑器中手动润色文本，并将关键修改反向同步给底层记忆系统。
- **US-08.02: 多模型智能路由 (P2)**
  - 作为系统管理员，我希望能将逻辑推演交给快速低成本模型（如 Qwen-2.5），将文本渲染交给长文本模型（如 Kimi/Claude），以便在保证质量的同时降低 API 成本。

## 4. 冲刺排期 (Sprint Roadmap)

基于现有产品规划与已批准的跨团队最终决策，未来 4 个 Sprint 的路线图如下。

> 说明：**Sprint 6 不是空档**。它仍然是 Release 2 的首个交付 Sprint，负责启动“看见世界”能力；Sprint 7 则承担 Release 2 的补齐与稳定性加固，然后再进入 Release 3 和 Release 4。

| Sprint | 版本目标 | 核心阶段 | 包含的 User Stories / 工作包 | 预期交付物 |
|---|---|---|---|---|
| **Sprint 6** | v0.6.0 | **Release 2: 看见世界（首个交付）** | US-05.02 (关系图谱)<br>US-05.03 (Agent 遥测)<br>Sprint 6 可视化资产管理计划 | 前端动态人物关系图谱；<br>后端输出关系演化数据；<br>基础遥测可透传。 |
| **Sprint 7** | v0.6.x | **Release 2: 看见世界（补齐）+ 稳定底盘** | 可视化补齐<br>统一 Agent 调用路径<br>GateKeeper 自愈循环<br>Telemetry SDK v1<br>流式稳定性改造 | 遥测日志面板；<br>统一调用链；<br>结构化关系数据；<br>可靠性护栏。 |
| **Sprint 8** | v0.7.x | **Release 3: 掌控世界** | US-06.01 (时间线分叉)<br>US-06.02 (多分支对比)<br>US-06.03 (节奏控制器) | 底层支持从历史节点分叉；<br>前端 Timeline 视图；<br>灰度发布与自动回滚。 |
| **Sprint 9** | v0.8.x | **Release 4: 创作作品（第一阶段）** | US-07.02 (交互式 Wiki)<br>US-08.01 (富文本编辑器)<br>US-08.02 (多模型路由)<br>长篇记忆与容量门禁 | 智能记忆系统；<br>多模型智能路由；<br>Wiki 与富文本编辑；<br>压测与 CI/CD 护栏。 |

> **演进逻辑说明**：
> 1. **Sprint 6** 先把“看见世界”的最小可视化能力立起来，让角色关系和推演过程不再是黑盒。
> 2. **Sprint 7** 不跳到新主题，而是继续补齐 Release 2，并把调用链、遥测和流式稳定性一次打牢，为后续分支能力提供地基。
> 3. **Sprint 8** 进入真正的差异化阶段，交付“时间线分叉”这一核心控制能力。
> 4. **Sprint 9** 再进入生产力阶段，优先补齐长篇一致性、成本优化和创作工作台，而不是一次性把 Release 4 所有事项全部铺开。
