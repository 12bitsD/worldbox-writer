# WorldBox Writer 产品深度思考

---

## 一、这东西到底在解决什么问题？

先说结论：**当前产品解决的问题和它实际提供的能力是错位的。**

README 写的是"解决 AI 写作中长文本一致性差和人类干预难以持久生效的痛点"。这句话没错，但它描述的是一个**工程问题**，不是一个**用户问题**。

用户不会说"我需要一个 DAG 状态机来保证叙事一致性"。用户会说：
- "我写到第 20 章角色性格崩了"
- "我不知道接下来该写什么"
- "我想看看如果主角做了不同选择会怎样"
- "AI 写的东西太水了，像在复读"

这四个问题，WorldBox Writer 的架构**理论上**都能解决。但它目前的实现**一个都没解决好**。

### 核心矛盾

产品的架构目标是做一个"故事推演引擎"，但用户需要的是一个"写作助手"。

推演引擎的隐喻是：Agent 自主推演，人类观察和干预。
写作助手的隐喻是：人类主导创作，AI 辅助执行。

**当前产品的姿态是"AI 是主角，人类是监工"。应该反过来："人类是作者，AI 是助手"。**

---

## 二、核心价值到底是什么？

排除所有花哨的功能，WorldBox Writer 有且只有**一个**真正的核心价值：

> **在 AI 生成长篇故事时，保持角色和世界的结构性一致性。**

这是它和 ChatGPT、Sudowrite、NovelAI 的本质区别。

- **ChatGPT/Claude**：写短文惊艳，写到第 5 章就开始忘记前面的设定，角色性格漂移。
- **Sudowrite**：有大纲和章节管理，但底层还是靠 prompt window 维持一致性，长了就崩。
- **NovelAI**：靠微调模型"隐式"维持一致性，黑盒，用户无法干预。
- **WorldBox Writer**：靠结构化数据（Scene Script / WorldState / Character Memory）"显式"维持一致性。角色行为先经过 Actor → Critic → GM 结算，再渲染成文字。

**这个"逻辑层先于文字层"的架构，是唯一真正不可替代的差异化。**

但问题是：这个差异化对用户来说是**隐形的**。用户看到的只是"输出的文字"，不会去理解底层的 DAG 和 Scene Script。

所以核心价值要**翻译成用户能感知的体验**：

| 底层能力 | 用户感知 |
|:---|:---|
| Scene Script 先于渲染 | "这 AI 写的长篇居然没有前后矛盾" |
| 角色隔离推演 | "每个角色的行为都像他自己，不是作者的提线木偶" |
| 分支系统 | "我可以探索不同剧情走向，然后选最好的" |
| 分层记忆 | "角色还记得第 3 章发生的事" |

## 三、哪些功能重要，哪些是工程师玩具

### 核心功能（必须做到极致）

1. **故事生成质量** — 当前最大短板。输出是模板复读，不是小说。
2. **干预体验** — 用户在关键时刻能自然地影响走向，不是输入命令行。
3. **角色一致性** — 这是核心差异化，必须让用户明显感知到。
4. **阅读体验** — 用户主要时间在"读"，不是在"看仪表盘"。
5. **导出** — 最终产物是可发布的文稿。

### 有用但不紧急

6. **分支/平行宇宙** — 概念很好，但当前用户量小时不重要。
7. **世界观设定管理** — 重要但应该简化，不需要单独一个面板。
8. **大纲/章节结构** — 长篇小说必需，但 MVP 阶段可以先做短篇。

### 工程师玩具（应该砍掉或隐藏）

9. **Telemetry 面板** — 没有用户想看 LLM 调用日志。
10. **Prompt Inspector** — 这是开发调试工具，不是产品功能。
11. **关系图谱可视化** — 看起来酷但没有实际用处（角色少的时候列表就够了）。
12. **双循环对比报告** — 纯工程产物。
13. **Diagnostics API** — 开发者工具。
14. **多模型路由/fallback** — 用户不关心底层用的什么模型。

### 应该砍掉的功能

15. **"推演日志"面板** — 这是把 Agent 内部状态暴露给用户，违反产品原则。
16. **T1/T2/T3 推演深度选择** — 用户不理解"tick"，应该叫"章节数"或干脆不暴露。
17. **"创世神"隐喻** — 虽然有创意但让产品显得像游戏而非工具。写作是严肃的创作活动。

## 四、交互应该怎么设计

### 当前问题

当前 UI 是一个**三栏仪表盘**：左侧设定集、中间故事正文、右侧运行状态。

这像什么？像 Grafana。像 Kubernetes Dashboard。像一个**运维监控系统**。

但用户是谁？是**写小说的人**。他们用的工具是什么？Word、Google Docs、Scrivener、Notion。

### 应该是什么样

**一个写作工具，不是一个推演仪表盘。**

核心界面应该像 Scrivener 或 Notion：
- 主区域是**富文本编辑器**，展示故事正文
- 右侧或底部有一个**轻量级的 AI 助手面板**
- 干预不是"输入命令"，而是"选择剧情方向"

### 具体设计

**默认视图：沉浸式阅读/写作**
```
┌─────────────────────────────────────────────┐
│  WorldBox Writer          [项目名]    [导出]  │
├─────────────────────────────────────────────┤
│                                             │
│         第一章：废墟中的相遇                    │
│                                             │
│    李青山站在倒塌的牌坊下，雨水顺着             │
│    他的斗笠滴落。他已经在这片废土上             │
│    走了三天，身上的干粮只剩最后一块。            │
│                                             │
│    "站住。"黑暗中传来一个女人的声音。           │
│                                             │
│    [AI 正在生成下一章...]                     │
│                                             │
│    ┌───────────────────────────────┐         │
│    │ 📖 接下来的故事走向：           │         │
│    │                               │         │
│    │ A. 李青山拔剑对峙，发现对方    │         │
│    │    是他失踪的师妹               │         │
│    │                               │         │
│    │ B. 对方是追杀他的门派长老，    │         │
│    │    他必须逃跑                   │         │
│    │                               │         │
│    │ C. [自己写...]                 │         │
│    │                               │         │
│    │ [跳过，让 AI 决定]             │         │
│    └───────────────────────────────┘         │
│                                             │
└─────────────────────────────────────────────┘
```

**关键设计原则：**

1. **主区域是文字，不是数据**。用户 90% 的时间在读/写，不是在看图表。
2. **干预是选择题，不是填空题**。当前的"输入自然语言干预指令"对普通用户太难了。应该提供 2-3 个 AI 生成的选项 + 一个"自己写"的入口。
3. **推演过程对用户透明但不打扰**。用户不需要知道"Director 正在生成 Scene Plan"，只需要看到"正在写下一章..."。
4. **设定集是参考，不是主界面**。折叠在侧边栏或抽屉里，需要时点开。
5. **编辑是第一公民**。用户应该能直接修改 AI 生成的文字，修改后 AI 基于修改后的内容继续。

### 干预机制重新设计

当前：每个节点暂停 → 用户输入文字干预 → 继续推演
问题：太频繁、太重、用户不知道写什么

应该改为：
- **默认不暂停**。AI 自动推演 3-5 章。
- **在剧情关键分歧点暂停**，提供选择题。
- **用户可以随时主动暂停**，输入自己的想法。
- **用户可以回退到任意章节**，从那里重新走另一条路。

这和互动小说（Visual Novel）的体验一致：读一段 → 做选择 → 继续读。

## 五、最小可行产品（MVP）定义

如果只能做 5 件事：

### 1. 修好输出质量（最高优先级）

当前的输出是不可用的。每一章都是模板复读。

根因分析：
- Narrator 拿到的输入是 Scene Script 的 objective 文本，不是结构化的事件列表
- LLM (gpt-4.1-mini) 可能只是在"扩写"objective 而非"创作小说"
- `estimated_completion_tokens: 0` 说明可能有 bug 导致 LLM 没有真正输出

修复方向：
- Narrator prompt 需要重写：输入应该是"场景描述 + 角色行动 + 对话"，而不是"围绕XX承接上一幕并制造新的选择"
- Scene Script 的 beats 需要包含具体的事件描述，而非抽象的目标
- 考虑换更强的模型做 Narrator（创意写作不适合用 mini 模型）

### 2. 修复 UI 交互（阻塞性 bug）

"开始推演"按钮不能点、"最近会话"打不开——这些是 P0 bug，不修无法测试任何东西。

### 3. 简化界面

砍掉 Telemetry、Inspector、推演日志面板。主界面就是一个富文本编辑器 + 一个轻量 AI 面板。

### 4. 重新设计干预体验

把"输入文字干预"改成"选择题 + 自由输入"。降低使用门槛。

### 5. 一键导出

用户最终要的是一个 .docx 文件。导出流程必须简单到"点一个按钮"。

## 六、差异化策略

### 应该走"写作工具"路线，不是"互动叙事"路线

原因：
1. **互动叙事市场太小**。AI Dungeon 已经证明了这个市场天花板低，用户留存差。
2. **写作工具市场大且付费意愿强**。Sudowrite 年收入千万美金级别。
3. **核心技术优势在写作场景更有价值**。"长篇一致性"对写小说的人是刚需，对玩互动叙事的人是 nice-to-have。

### 与竞品的定位差异

| 竞品 | 定位 | WorldBox Writer 的差异定位 |
|:---|:---|:---|
| ChatGPT | 通用对话式写作 | "专门写长篇小说，不会写到后面忘了前面" |
| Sudowrite | 章节级写作助手 | "能帮你规划整本书的结构，不只是改一段文字" |
| NovelAI | 模型驱动的自由写作 | "你能控制故事走向，不只是让 AI 自由发挥" |
| Novelcrafter | 大纲+章节管理 | "不只管大纲，AI 真的能帮你写，而且角色不会崩" |

### 一句话定位

> **WorldBox Writer：写长篇小说时，AI 不会忘记前面写了什么。**

简洁、具体、可验证。比"沙盒式 AI 小说生成系统"好 10 倍。

## 七、产品路线图建议

### Phase 0：能用（2 周）
- 修好 UI 交互 bug
- 修好输出质量（重写 Narrator prompt，换模型）
- 修好 initializing 卡住问题

### Phase 1：好用（1 个月）
- 重新设计主界面（富文本编辑器为主）
- 选择题式干预（替代文字输入）
- 角色名用 LLM 生成真实人名
- 章节化组织（替代 tick/node 概念）
- 一键导出 DOCX

### Phase 2：想用（2 个月）
- 用户编辑 AI 文本后继续推演
- 大纲导入
- 模板市场（武侠/科幻/言情等预设）
- 进度保存/恢复的明确 UI

### Phase 3：离不开（3 个月+）
- SaaS 部署
- 长篇小说的完整章节管理
- 角色弧线追踪
- 多风格切换
- 社区分享

## 八、最终判断

WorldBox Writer 的**技术架构是好产品**，但**当前实现不是好产品**。

它像一辆装了 F1 引擎的拖拉机——引擎很强，但车身、轮胎、方向盘都需要换。

核心问题不是"功能不够多"，而是：
1. **输出质量不可用**（最致命）
2. **用户界面像运维工具**（劝退目标用户）
3. **干预体验太重**（用户不知道怎么用）

好消息是：这三个问题都是"换壳"问题，不是"换引擎"问题。底层架构（DAG + 角色隔离 + Scene Script）是正确的，不需要推倒重来。

**最该做的一件事：把 Narrator 的输出质量拉到"可读的小说"水平。** 一旦用户看到 AI 写出了像样的第一章，其他所有问题都有了解决的动力。
