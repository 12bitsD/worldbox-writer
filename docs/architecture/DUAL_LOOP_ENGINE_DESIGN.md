# WorldBox Writer: 双循环推演引擎与一键成书架构方案

本文档旨在为“高并发角色独立推演”与“长篇网文一键生成”提供架构蓝图。
文档遵循 **“第一性原理”** 进行设计拆解，剔除过度沙盒化的技术自嗨，确保每一项架构决策都严格服务于最终目标：**以极低的幻觉率和高可控性，生成连贯、涌现、具备网文张力的长篇小说。**

---

## 1. 终极目标与期望 (Core Objectives & Expectations)

在设计任何技术模块前，必须明确系统要达成什么效果，以及解决什么痛点。

### 1.1 期望一：彻底消灭“串戏”与“逻辑崩坏”
*   **目标**：角色在做决策时，绝对不能知道只有读者或其他角色才知道的秘密。长篇推演 100 章后，逻辑依然严密。
*   **解决手段（原理验证）**：
    *   **推演与渲染解耦**：大模型在生成华丽长文时极易产生幻觉（例如描写打斗时凭空捏造出一把武器）。我们将系统分为“只生成精简 JSON 意图（如：拔剑、逃跑）”的逻辑层，和“只负责把 JSON 扩写为小说”的渲染层。逻辑层由于 Token 极短且结构化，出错率将呈指数级下降。
    *   **物理级上下文隔离**：不再将所有角色的状态打包放入同一个大 Prompt。系统为场景中的每个核心角色独立拉起一个 Agent 实例。角色 A 的 Prompt 中只注入 A 自己的属性卡、A 自己视角的历史记忆片段，以及当前场景的公开信息。这在物理层面上杜绝了角色“偷看剧本”的可能。

### 1.2 期望二：拒绝“流水账”，确保“网文张力”
*   **目标**：多 Agent 沙盒极易陷入无聊的日常寒暄。我们要的是跌宕起伏的网文，不是模拟人生。
*   **解决手段（原理验证）**：
    *   **引入“叙事压力 (Narrative Pressure)”与导演强干预**：系统中的 Director Agent 不仅仅是一个环境描述者，它实质上是一个“宏观情节控制器”。它内部维护着一条设定的情节曲线（如：平缓 -> 危机 -> 高潮）。当发现沙盒演化陷入平淡时，Director 会在下发给 Actor 的私有 Context 中隐蔽地注入心理暗示（例如强制注入潜意识：“你突然感到莫名的烦躁，急需寻找一个发泄口”），从而以符合人物设定的方式，强行打破沙盒的僵局，推动冲突发生。

### 1.3 期望三：所见即所得的调优与交互体验
*   **目标**：告别黑盒等待。用户必须能实时看到推演过程，且能随时透视和修改任何角色的底层设定。
*   **解决手段（原理验证）**：
    *   **基于 SSE 的并发流式推送**：利用 Server-Sent Events 技术，任何一个 Actor Agent 思考完毕其意图（JSON）后，立刻推送到前端并渲染在日志流中。用户无需等待所有角色思考完毕或小说渲染完成，即可实时观察沙盒动态。
    *   **Agent Inspector 透视调试**：提供一个统一的调试收口。用户点击界面上的任何角色，即可看到该角色当前 LLM 调用的完整 Prompt（包括它是如何被组装的，以及召回了哪几条具体的历史记忆）。通过这种“白盒化”设计，开发者或高阶用户可以精准定位逻辑跑偏的原因（是记忆没召回，还是人设被污染），并直接在面板中修改设定进行热重载。

## 2. 核心架构设计：双循环与对抗结算 (Architecture Design)

为实现上述目标，系统重构为 **“内循环推演 (逻辑)”** 与 **“外循环渲染 (文笔)”**。

### 2.1 内循环：强导向并发推演 (Directed Spotlight Inference)
*   **设计期望**：快速生成逻辑严密、充满冲突的结构化“场景剧本 (Scene Script)”。

**【核心组件与机制】**

1.  **Director Agent (带叙事压力的场控)**
    *   **期望目的**：掌控剧情节奏，防止沙盒流水账化。
    *   **运行机制**：Director 不仅决定下一幕的地点和出场人物（打聚光灯），更重要的是，它会隐蔽地向被召回的角色注入 **“叙事压力 (Narrative Pressure)”**。例如，系统当前处于“高潮前夕”，Director 会在主角的 Prompt 中附加指令：“你现在内心感到极度不安，迫切想要打破僵局”。
2.  **Actor Agent (按需唤醒与绝对隔离)**
    *   **期望目的**：实现真正的“信息隔离”和群像涌现，同时控制 Token 成本。
    *   **运行机制**：系统只为 Director 选定的关键角色（或被点化的关键物品）实例化并行的 Actor Agent。通过 Map-Reduce 并发调用，每个 Actor 只能读取自己的“私有记忆库”和“公开环境简报”，独立做出反应。
3.  **LLM Critic & GM Agent (对抗式逻辑结算)**
    *   **期望目的**：拦截离谱的行动意图，保证世界观设定的物理/魔法法则不被破坏。
    *   **运行机制**：放弃复杂的硬编码规则引擎。Actor 提交意图后，先经过廉价的 Critic Agent 审查（判断是否符合世界观设定）。审查通过后，再由 GM Agent 进行冲突结算，输出本轮唯一的**《客观事件日志 (Scene Script)》**。

### 2.2 多层认知记忆管线 (Cognitive Memory Stream)
*   **设计期望**：解决长篇连载中的“遗忘”和“记错设定”问题，同时降低每次调用的上下文长度。
*   **运行机制**：
    *   角色的记忆分为三层：**工作记忆**（最近发生的事，直接入 Prompt）、**情景记忆**（历史日志，按重要性+近期性+相关性召回）、**反思记忆**（后台异步触发，将多次经历总结为一条高级性格设定，如“经历三次被骗 -> 生性多疑”）。

### 2.3 外循环：小说渲染管线 (Narrative Rendering Loop)
*   **设计期望**：将干瘪的结构化剧本，转化为大师级文笔的长篇网文。
*   **运行机制**：
    *   **Narrator Agent** 作为一个后台监听器。当内循环积累够了一个完整的情节片段（Scene），触发高文笔模型（如 Claude 3.5），结合指定的主角视角和文风设定，将其流式渲染为 1000-2000 字的小说正文（Novel Chapter）。

## 3. 底层基建支撑 (Infrastructure Support)

为了让上述逻辑顺畅且可观测地运行，必须夯实以下基建：

### 3.1 高并发与实时响应 (Concurrency & SSE)
*   **目的**：推演耗时不能随出场人数增加而变慢；用户体验必须像看群聊一样实时。
*   **手段**：Actor 思考阶段采用异步并发。任何 Actor 一旦思考完毕，立刻通过 Server-Sent Events (SSE) 管道推送到前端。

### 3.2 深度可观测性与调优收口 (Observability & Tuning)
*   **目的**：让开发者和高阶用户能直观地调试“为什么系统会写出这行字”。
*   **手段**：
    *   提供 Agent Inspector 透视面板。用户可以点击任何角色，直接查看它当前被注入了什么“叙事压力”、召回了哪些“历史记忆片段”。
    *   所有 Prompt 模板外部化，支持界面热更新，实现真正的“创世神控制台”。

---

## 4. 架构设计的价值收敛 (Value Alignment)
*   **为什么这么设计？** 因为我们要在“纯粹失控的沙盒”和“死板线性的脚本”之间找到最佳平衡点。
*   **双循环** 保证了逻辑的严丝合缝与最终文笔的华丽；**带压力的 Director** 保证了网文的爽感和节奏；**隔离并发的 Actor** 贡献了意想不到的群像涌现；**Critic 对抗结算** 则守住了不崩坏的底线。
*   这套基于第一性原理推演出的架构，才是真正面向“长篇网文生产力”的终极解法。
