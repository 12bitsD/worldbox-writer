## 2. 推荐开发顺序

建议按以下顺序推进，避免前后返工：

1. `S7-EN-01`
2. `S7-EN-02`
3. `US-05.06`
4. `US-05.05`
5. `US-05.04`
6. `S7-EN-03`
7. `US-02.04`
8. `S7-EN-04`

原因：

- 先把调用链元数据和 telemetry 契约补齐，再做日志 UI，避免前端先做一版假字段。
- 先修一致性，再做图谱和日志体验，否则 UI 会建立在不稳定状态上。
- 性能优化与自愈属于后置项，不应阻塞 Observe 主目标收口。

## 3. 需求拆解

## 3.1 S7-EN-01 调用链元数据统一化

**目标**

- 让所有 Agent 调用在同一条抽象路径下产出统一元数据，为 Telemetry 关联和后续多模型路由打底。

**期望收口**

- 所有经 `chat_completion()` 发起的调用都能稳定拿到同一套元数据。
- 元数据最少包含 `provider`、`model`、`request_id`、`role`。
- 不引入第二套 LLM 调用入口，不做额外的大重构。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S7-EN-01-T1 | 盘点现有 Agent 调用入口，确认全部仍走 `chat_completion()` | AI 系统团队 | 无 | 调用清单与例外项 |
| S7-EN-01-T2 | 为 `chat_completion()` 增加统一返回元数据结构或 side-channel | AI 系统团队 | T1 | 统一调用元数据对象 |
| S7-EN-01-T3 | 在 Actor / Director / GateKeeper / Narrator / WorldBuilder / NodeDetector 调用点接入元数据 | AI 系统团队 | T2 | 统一接入代码 |
| S7-EN-01-T4 | 为元数据新增 L1 测试，覆盖 provider/model/request_id 回传 | AI 系统团队 | T2 | 单元测试 |
| S7-EN-01-T5 | 更新 `utils/llm.py` 与相关架构说明 | AI 系统团队 | T4 | 文档更新 |

**验收检查**

- 任一 Agent 调用都能在日志或测试中看见一致的 provider/model/request_id。
- 代码搜索后不存在绕过该路径的新硬编码 LLM 调用。

## 3.2 S7-EN-02 Telemetry Correlation Schema v1

**目标**

- 把当前“可读事件”升级为“可关联事件”，能串起一次请求、一轮推演和一组前端展示。

**期望收口**

- `TelemetryEvent` 字段冻结并文档化。
- 后端持久化、API 返回、SSE 增量三条链路字段一致。
- 前端类型能直接消费，无需自行猜字段。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |

| S7-EN-02-T2 | 更新 `TelemetryEvent` 后端模型与序列化逻辑 | 服务端架构团队 | T1 | 模型与序列化代码 |
| S7-EN-02-T3 | 更新 API / SSE / DB round-trip，保证字段全链路一致 | 服务端架构团队 | T2 | 存储与接口代码 |
| S7-EN-02-T4 | 更新前端 `frontend/src/types` 与 fixtures | 前端团队 | T3 | TS 类型与 fixtures |
| S7-EN-02-T5 | 新增契约测试，覆盖 GET/SSE/恢复场景 | 服务端架构团队 + 前端团队 | T3, T4 | 后端 API 测试与前端 fixture 测试 |

**验收检查**

- 同一个 `event_id` 在 DB、REST、SSE、前端 state 中字段一致。
- 新增字段进入 fixture 和测试，不再只是隐式约定。

## 3.3 US-05.06 实时、历史与刷新恢复一致性

**目标**

- 解决“实时看到一套，刷新后又是另一套”的状态漂移问题，让 Observe 面板可信。

**期望收口**

- 同一会话从实时进入、刷新恢复、最近会话打开三条路径进入时，节点、telemetry、world 数据一致。
- 节点和事件按稳定 ID 幂等合并，不重复、不丢失、不乱序。
- 至少有一条自动化回归测试覆盖完整恢复链路。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| US-05.06-T1 | 画出现有 GET + SSE + polling + session restore 的状态合并路径 | 前端团队 + 服务端架构团队 | S7-EN-02 | 合并路径图 |
| US-05.06-T2 | 定义节点、telemetry、world 的幂等合并规则 | 前端团队 + 服务端架构团队 | T1 | 合并规则说明 |
| US-05.06-T3 | 修正 `useSimulation` 中的增量合并逻辑，统一以稳定 ID 去重 | 前端团队 | T2 | Hook 改造 |
| US-05.06-T4 | 修正服务端初始载荷和增量事件的字段时机，避免先后不一致 | 服务端架构团队 | T2 | API/SSE 修复 |
| US-05.06-T5 | 为“启动 -> 收到事件 -> 刷新 -> 恢复 -> 历史打开”补自动化测试 | 前端团队 + 服务端架构团队 | T3, T4 | API/UI 回归测试 |
| US-05.06-T6 | 整理最近会话和恢复语义，更新 README / Runbook 中的行为描述 | 前端团队 + 服务端架构团队 | T5 | 文档更新 |

**验收检查**

- 打开同一 `sim_id` 时，`nodes.length`、`telemetry.length`、world 关键字段在三条路径一致。
- 不再出现刷新后少边、重复事件、最后一条 narrator 文本丢失。

## 3.4 US-05.05 遥测日志筛选、分组与关联阅读

**目标**

- 让 Telemetry 面板从“纯事件列表”变成“可阅读、可定位问题的调用链视图”。

**期望收口**

- 用户可按 `agent` 和 `stage` 过滤。
- 用户可按 `tick` 或 `agent` 分组阅读。
- 事件卡片能展示关联字段，至少能看出一条调用链上下文。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| US-05.05-T1 | 确认日志阅读模型：默认排序、分组维度、过滤交互 | 前端团队 + 产品 | S7-EN-02 | 交互草图 |
| US-05.05-T2 | 为 Telemetry 面板增加过滤状态与分组派生逻辑 | 前端团队 | T1 | 组件状态逻辑 |
| US-05.05-T3 | 事件卡片增加关联元数据显示，不直接铺满原始 JSON | 前端团队 | S7-EN-02 | 卡片 UI |
| US-05.05-T4 | 新增 fixtures 覆盖多 agent、多 stage、多 tick 的混合日志 | 前端团队 | T2 | fixtures |
| US-05.05-T5 | 为过滤、分组、空状态、历史状态补前端测试 | 前端团队 | T2, T3, T4 | Vitest 测试 |
| US-05.05-T6 | 视需要补服务端 stage/agent 枚举规范，避免前端读到脏数据 | 服务端架构团队 | T1 | 字段约束修正 |

**验收检查**

- 过滤任一 `agent` 或 `stage` 后，列表只显示匹配事件。
- 分组切换不会改变事件总数，只改变阅读方式。
- 关联字段展示可读，不再直接依赖 `JSON.stringify(payload)` 作为主展示。

## 3.5 US-05.04 关系图谱交互与聚焦

**目标**

- 让关系图谱从“静态示意图”升级成“可定位角色关系变化”的交互面板。

**期望收口**

- 支持节点选中、高亮相邻边、角色聚焦。
- 支持边详情查看，展示关系标签、强度、备注和最后更新时间。
- 历史会话恢复后图谱行为一致，不依赖前端重新推理关系。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| US-05.04-T1 | 确认当前 structured relationship 是否已覆盖交互所需字段 | AI 系统团队 + 前端团队 | 无 | 字段检查清单 |
| US-05.04-T2 | 为图谱组件加入选中节点、聚焦节点、悬停边等交互状态 | 前端团队 | T1 | 组件状态逻辑 |
| US-05.04-T3 | 增加边详情面板或详情卡，展示 `label`/`affinity`/`note`/`updated_at_tick` | 前端团队 | T2 | 详情 UI |
| US-05.04-T4 | 若字段不足，补服务端序列化与前端类型 | AI 系统团队 + 前端团队 | T1 | 序列化/类型修正 |
| US-05.04-T5 | 新增 fixtures 与交互测试，覆盖空图、单边、多边、历史恢复 | 前端团队 | T2, T3, T4 | 测试与 fixtures |
| US-05.04-T6 | 调整文案与空状态，保证在 0 边/少角色场景下仍可解释 | 前端团队 | T3 | UI 打磨 |

**验收检查**

- 点击任一角色后，只高亮其一跳关系，其他节点弱化。
- 边详情信息来自后端数据，不是前端硬编码映射。
- 恢复历史会话后，交互行为与实时会话一致。

## 3.6 S7-EN-03 最小可靠性护栏

**目标**

- 给当前流式链路补最必要的失败策略，避免一处抖动把 Observe 体验整体拖垮。

**期望收口**

- 前端 SSE 失败能稳定回退到 polling。
- 增量事件能幂等合并，不因重连造成重复。
- 后端对关键长链路有基本 timeout / error 收敛策略。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S7-EN-03-T1 | 梳理流式主链路失败点：SSE 断开、慢响应、重复事件、恢复中断 | 服务端架构团队 + 前端团队 | US-05.06 | 风险清单 |
| S7-EN-03-T2 | 为前端重连和 polling fallback 加入显式状态与幂等保护 | 前端团队 | T1 | Hook 改造 |
| S7-EN-03-T3 | 为服务端关键调用点补 timeout / 异常收口日志 | 服务端架构团队 | T1 | 后端护栏代码 |
| S7-EN-03-T4 | 增加失败场景测试：SSE 中断、重复推送、恢复后继续读取 | 前端团队 + 服务端架构团队 | T2, T3 | 回归测试 |
| S7-EN-03-T5 | 在 Runbook 中记录故障诊断路径 | 服务端架构团队 | T4 | Runbook 更新 |

**验收检查**

- SSE 人为断开后，前端能自动降级，不出现白屏或整页失效。
- 重连后重复事件不会造成列表重复增长。

## 3.7 US-02.04 GateKeeper 拒绝自愈

**目标**

- 在不牺牲边界控制的前提下，减少低质量拒绝带来的空转。

**期望收口**

- 被拒绝候选可按 `revision_hint` 自动重试有限次数。
- Telemetry 里能看见“被拒绝 -> 修正 -> 重新校验 -> 最终结果”的完整链路。
- 失败时仍然明确失败，不做隐式吞错。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| US-02.04-T1 | 审查当前 `revision_hint` 质量和稳定性，确认是否可驱动重试 | AI 系统团队 | 无 | 可行性结论 |
| US-02.04-T2 | 设计重试策略：最大次数、何时停止、如何避免自激循环 | AI 系统团队 | T1 | 重试规则 |
| US-02.04-T3 | 在 `gate_keeper_node` 周边实现修正重试与 telemetry 发射 | AI 系统团队 | T2, S7-EN-02 | 后端逻辑 |
| US-02.04-T4 | 增加 L1/L2 测试，覆盖一次成功修正和超过上限失败 | AI 系统团队 | T3 | 单测与集测 |
| US-02.04-T5 | 更新架构说明，明确该机制是“自愈”而不是“绕过约束” | AI 系统团队 | T4 | 文档更新 |

**验收检查**

- 至少一个固定场景能复现自动修正成功。
- 重试次数耗尽后，tick 行为可解释，日志完整。

## 3.8 S7-EN-04 长列表性能改造

**目标**

- 在真实性能压力下，保证长会话仍可交互；如果当前实现足够，则不做过度工程。

**期望收口**

- 先有性能基线，再决定是否引入虚拟列表或查询缓存。
- 若引入新依赖，必须给出收益和替代方案说明。
- 不为了“技术正确”提前吞入沉重框架。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S7-EN-04-T1 | 以 fixture 或脚本构造大节点量/大日志量场景，建立性能基线 | 前端团队 | US-05.05, US-05.06 | 基线数据 |
| S7-EN-04-T2 | 评估当前 `useSimulation` + 现有组件是否已成为瓶颈 | 前端团队 | T1 | 评估结论 |
| S7-EN-04-T3 | 若性能不足，再选择虚拟列表或缓存方案并做最小接入 | 前端团队 | T2 | 优化实现 |
| S7-EN-04-T4 | 为优化前后做对比记录，沉淀到 Sprint 复盘 | 前端团队 | T3 | 对比报告 |

**验收检查**

- 没有基线数据前，不启动依赖引入。
- 若已优化，需能量化说明收益。

## 6. 总收口标准

Sprint 7 是否真正收口，不看“任务都动过”，而看下面 5 条是否同时成立：

1. 图谱交互已从静态展示升级为可聚焦、可解释。
2. Telemetry 已从纯列表升级为可筛选、可分组、可关联阅读。
3. 同一会话在实时、历史、刷新三种进入方式下结果一致。
4. 至少一条完整链路可从 Agent 调用元数据追到前端事件展示。
5. P0 故事的测试、文档、演示链路全部闭环。
