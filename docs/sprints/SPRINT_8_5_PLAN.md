# Sprint 8.5 计划

**文档状态**：Delivered  
**适用版本**：Sprint 8.5 / v0.7.x follow-up  
**最后更新**：2026-04-19

本文档用于承接 Sprint 8 已交付的 branching 能力，并聚焦解决另一个已经暴露出来的体验问题：用户发起第一次推演后，到看到第一条可感知反馈之间等待过长。

---

## 1. 这份 Sprint 8.5 在解决什么问题

Sprint 8 解决了“能分叉、能切换、能续跑”的控制闭环，但并没有直接解决“第一次可见反馈很慢”的问题。

当前链路里已经有 token 级流式正文能力，但它只发生在 `Narrator` 阶段；而 `Narrator` 之前还要串行经过：

1. `Director` 初始化世界
2. `WorldBuilder` 扩写世界
3. `Actor` 生成候选事件
4. `GateKeeper` 校验候选事件
5. `NodeDetector` 固化节点
6. `Narrator` 才开始流式出字

这意味着“渐进出字”能力虽然存在，但首个 token 仍然要等多轮前置 LLM 调用完成后才出现。用户体感上的问题不是“完全没有流式”，而是“流式来得太晚”。

---

## 2. Sprint Goal

Sprint 8.5 的目标不是重做整套服务架构，而是把“首个可见反馈”拉到足够早，让用户在第一次推演的等待过程中始终知道系统正在做什么，并尽可能更早看到第一幕正文。

---

## 3. 业务结果

- 用户在启动推演后，能尽快看到初始化阶段的实时进度，而不是长时间停留在单一 loading 文案。
- 第一幕正文的流式输出应尽量提前开始，不再被完整世界扩写阻塞在最前面。
- 系统在保持当前单体后端结构的前提下，先把“首屏体验”做顺，再评估是否值得进一步拆分成独立 worker / queue 形态。

---

## 4. 关键判断

### 4.1 这轮不优先拆 AI 微服务

从代码现状看，当前瓶颈首先是**关键路径过长**和**首屏进度暴露太弱**，而不是“HTTP 进程和 LLM 编排必须立刻分家”。

如果现在直接把 LLM 编排拆成独立 AI 服务，会带来：

- 一次额外网络跳转和状态同步复杂度
- 新的任务队列、重试、幂等和会话恢复问题
- SSE / 轮询 / 会话状态真相源重新设计

但它并不会自动减少 `Director -> WorldBuilder -> Actor -> GateKeeper -> NodeDetector -> Narrator` 这条串行链路本身的耗时。

所以 Sprint 8.5 的判断是：

- **短期正确解**：先优化当前单体架构中的关键路径和用户反馈方式。
- **中期更优组织形式**：保留未来演进到 `API / Session Service + Simulation Worker` 的方向，但不把它当成本期前置。

### 4.2 这轮真正要打的不是“总耗时”，而是“首个可见反馈时间”

用户第一次使用时，最容易流失的不是总 20 秒还是 24 秒，而是前 5-10 秒里看起来像“没有开始工作”。

所以 Sprint 8.5 的主指标不是绝对总时长，而是：

- 首个进度事件是否能尽早出现在 UI
- 首个正文 token 是否比现在更早出现
- 用户是否能理解系统当前在做哪一步

## 6. 设计决策

### 6.1 先显示进度，再等世界完整

当前后端在 `world` 尚未挂到 session 上时，前端很容易只能显示“正在初始化”。Sprint 8.5 的第一决策是：**初始化阶段的 Telemetry 必须先成为一等公民**。

这意味着：

- `GET /simulate/{id}` 在 `world is None` 时也要返回已有 telemetry
- 前端在 `POST /start` 之后要优先连上 SSE，再做首个状态对账
- loading UI 不能只是一句固定文案，而要反映当前阶段

### 6.2 首个正文优先于完整世界扩写

`WorldBuilder` 提升世界丰富度，但它不是生成第一幕的绝对前置。Sprint 8.5 的第二个决策是：**把 `WorldBuilder` 从首个正文 token 的关键路径里移开**。

推荐组织方式：

- `Director` 完成后，直接进入第一轮 `Actor -> GateKeeper -> NodeDetector -> Narrator`
- 第一幕开始流式输出之后，再补 `WorldBuilder`
- `WorldBuilder` 依旧保留为后续世界细节和长期质量的来源，但不阻塞第一幕出现

### 6.3 先做单体内的后台任务优化，不立刻拆微服务

Sprint 8.5 默认继续沿用“FastAPI + 线程池 + 内存 session + SQLite”的本地架构。

这轮的目标是先验证一个更务实的命题：

> 如果首屏体验问题主要来自关键路径顺序和反馈暴露不足，那么在不拆服务的情况下，是否已经能把体验改善到可接受。

只有当这一轮完成后仍证明体验明显不足，才进入下一阶段：

- 独立 `Simulation Worker`
- 队列化任务提交
- API 层只负责会话与流式事件聚合

---

## 7. 为什么这套方案大概率有效

这份方案并不依赖“模型突然更快”，它依赖的是两个更稳的改进：

1. **把已有进度更早展示出来**  
   当前系统已经会发 `world_initialized`、`world_enriched`、`proposal_generated`、`node_committed`、`narrator.started/completed` 等事件。把这些事件更早暴露到 UI，本质上就是把“无反馈等待”变成“可理解等待”。

2. **减少首个 token 之前必须完成的步骤数**  
   只要 `WorldBuilder` 不再卡在第一幕正文之前，首个 token 的关键路径就至少少掉一轮独立 LLM 调用。这是结构性缩短，不依赖特定 provider 的偶然波动。

所以 Sprint 8.5 的预期不是“总耗时必然腰斩”，而是：

- 首个进度反馈明显提前
- 首个正文 token 明显提前
- 用户对等待过程的理解成本显著下降

---

## 8. 非目标

Sprint 8.5 默认不承诺以下事项：

- 拆分独立 AI 微服务
- 引入 Redis / Kafka / Celery 等分布式队列
- 对所有 Agent 做并行调用重构
- 引入 provider 级自动 benchmark / 自动路由系统
- 重做完整前端视觉系统

这些方向可能有价值，但都不应先于“把第一次等待体验做顺”。

---

## 9. Sprint 退出标准

- 启动新推演后，在首个 `world` 快照之前，前端已经能显示至少一条初始化阶段进度事件。
- 第一幕正文开始流式输出前，不再被完整 `WorldBuilder` 阶段阻塞。
- 前端主区域存在可读的阶段进度展示，而不是只在右侧 Telemetry 面板里藏事件。
- 默认验证通过：`make lint`、`make test`。
