## 1. 这份计划解决什么问题

这份计划用于把已批准的最终决策改写成可执行的 Agile Sprint 计划，明确 Sprint 7-9 的目标、范围、团队分工、验收标准与风险边界。

### 核心排期原则

1. **产品目标优先**：不做脱离用户价值的纯技术重构，技术改造只作为功能交付的护栏。
2. **里程碑连续推进**：Sprint 6 建立“看见世界”的最小闭环，Sprint 7 补齐其稳定性与体验，Sprint 8 解决“掌控世界”，Sprint 9 解决“创作作品”。
3. **每个 Sprint 都必须可演示**：每轮结束都要有前后端联调可见的增量，而不是只完成底层准备。

## 3. Sprint 7 计划

**当前状态**：Delivered on 2026-04-18

### Sprint Goal

Sprint 7 的目标不是从零开始做“看见世界”，而是在 Sprint 6 最小闭环已经建立的前提下，补齐可读性、稳定性和统一调用链。

### 业务结果

- 用户能获得**更完整、更易读的关系图谱体验**。
- 用户能获得**可筛选、可关联的 Agent 遥测日志**。
- 系统在**大量节点流式推演**时保持可用，不因前端渲染或接口抖动失稳。

### Sprint Backlog 组织方式

Sprint 7 不再把所有事项都堆成同级 P0，而是拆成两层：

- **承诺交付的用户故事（P0）**：直接面向“看见世界”的用户价值。
- **支撑这些故事的工程使能项（P0/P1）**：只做与 Sprint Goal 强相关的底盘工作。

这样组织的目的，是避免把“用户可见能力”和“技术准备动作”混写成 8 个并列 P0，导致 Sprint Goal 被工程事项稀释。

## 4. Sprint 8 计划

### Sprint Goal

Sprint 8 的目标是交付“时间线分叉”这个核心差异化能力，让用户真正能改写世界走向，而不是只能被动观看。

### 业务结果

- 用户能从任意历史节点创建新分支。
- 用户能在前端查看并切换多条世界线。
- 系统能对高风险分支功能做灰度控制与快速回滚。

### Sprint Backlog

| ID | 条目 | 优先级 | 负责人 | 说明 | 验收结果 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| S8-01 | 历史节点分叉能力 | P0 | AI 系统团队 | 实现 `fork_at_node()`，支持从任意节点派生平行世界线 | 新分支能继承父节点上下文 |
| S8-02 | 多 Actor 协商机制 | P0 | AI 系统团队 | 支持多个角色提议冲突、仲裁与推进 | 冲突场景可复现并产出稳定结果 |
| S8-03 | 叙事节奏控制器 | P0 | AI 系统团队 | 接入用户节奏控制参数，影响后续推演 | 节奏参数能改变提议筛选结果 |
| S8-04 | 多分支时间线视图 | P0 | 前端团队 | 开发 Timeline 组件，支持分支切换与层级展示 | 用户可查看主线与支线关系 |
| S8-05 | 前端运行时状态校验 | P0 | 前端团队 | 使用 Zod 对多分支状态做强校验，降低异常数据导致的白屏 | 异常状态被拦截并可提示 |
| S8-06 | 灰度发布与自动回滚 | P0 | 服务端架构团队 | 为分支功能提供灰度开关和回滚能力 | 发布失败时可快速回退 |
| S8-07 | 端到端 Trace 覆盖 | P0 | 服务端架构团队 | 覆盖多分支链路，支持线上问题定位 | 每条分支请求可追踪上下文 |

### 架构预留：分支合并（Reconvergence）

当前 `StoryNode` 已预留 `branch_id` 和 `merged_from_ids` 字段，`WorldState` 已预留 `branches` 注册表和 `active_branch_id`。Sprint 8 在实现分叉时，必须同时验证这些预留字段的序列化/反序列化兼容性，为未来的分支合并操作（类似 Git merge）留出架构空间。本期不实现合并功能，但 `fork_at_node()` 的实现必须正确填充 `branch_id` 并注册新分支到 `WorldState.branches`。

### 状态爆炸防范措施

随着分支数量增加，前端状态树和后端节点存储会急剧膨胀。Sprint 8 必须落实以下防范措施：

1. **前端虚拟化**：S8-04 的多分支时间线视图必须采用虚拟滚动（Virtual Scroll），只渲染视口内可见的节点。非活跃分支默认折叠，仅在用户展开时加载详细节点数据。
2. **后端懒加载**：`GET /api/simulate/{id}` 默认只返回 `active_branch_id` 对应的节点列表。新增 `?branch=all` 参数时才返回全量节点。
3. **分支数量上限**：单个会话的分支数量暂定上限为 10 条。超出时提示用户归档或删除不活跃分支。
4. **前端状态校验**：S8-05 的 Zod 校验必须覆盖分支数据结构，拦截因分支膨胀导致的异常状态。

### 关键依赖

- S8-04 依赖 S8-01 提供稳定的分支数据结构。
- S8-03 依赖 Sprint 7 已统一的调用链与遥测日志。
- S8-06 与 S8-07 是分支功能上线的发布前置条件。
- S8-01 依赖 `models.py` 中已预留的 `branch_id`、`merged_from_ids`、`branches`、`active_branch_id` 字段。

### Sprint 退出标准

- 用户能在真实界面中对同一历史节点创建并切换至少 2 条分支。
- 多分支状态在接口、前端内存与日志链路中保持一致。
- 分支功能可以灰度发布，并具备自动回滚预案。
- 非活跃分支在前端默认折叠，不影响主线性能。
- 单会话分支数量超过上限时有明确提示。

## 5. Sprint 9 计划

### Sprint Goal

Sprint 9 的目标是补齐长篇创作所需的记忆、成本与编辑能力，让产品从“可玩”走向“可长期使用”。

### 业务结果

- 长篇推演的一致性明显提升，不再频繁遗忘设定。
- 模型调用成本可控，逻辑推演与文本渲染按场景分流。
- 用户能在 Wiki 与富文本编辑器中继续整理和打磨内容。

### Sprint Backlog

| ID | 条目 | 优先级 | 负责人 | 说明 | 验收结果 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| S9-01a | 记忆持久化 + LLM 摘要归档 | P0 | AI 系统团队 | 将内存记忆迁移至 SQLite，引入 LLM 摘要归档机制，当记忆条目超过阈值时自动压缩旧记忆为摘要 | 旧记忆被压缩后仍可被召回，新记忆重启后不丢失 |
| S9-01b | 向量检索集成 (ChromaDB) | P1 | AI 系统团队 | 在 S9-01a 稳定后，引入 ChromaDB 做语义检索，替换当前的 TF-IDF 简单匹配 | 语义相关的设定能被召回，即使关键词不完全匹配 |
| S9-02 | 多模型智能路由 | P0 | AI 系统团队 | 逻辑层走低成本模型，创意层走高质量模型。必须同时交付多模型评估基准测试（见下方多模型测试策略） | 成本统计可见，路由规则可配置，低成本模型不导致逻辑降级 |
| S9-03 | 交互式设定 Wiki | P0 | 前端团队 | 支持世界设定的查看、编辑、保存与校验 | 设定改动可落库并回显 |
| S9-04 | 富文本编辑器集成 | P0 | 前端团队 | 按需加载编辑器组件，支持长文润色 | 长文编辑体验可用且不卡顿 |
| S9-05 | 端侧草稿恢复 | P0 | 前端团队 | 接入 IndexedDB 定时快照，支持异常关闭恢复 | 断电或误关闭后可恢复草稿 |
| S9-06 | 压测与容量门禁 | P0 | 服务端架构团队 | 建立主链路压测基线，识别容量上限 | 压测报告可复用且纳入门禁 |
| S9-07 | CI/CD 提速与线上诊断闭环 | P0 | 服务端架构团队 | 完善质量门禁与线上诊断工具链 | 发布、回归、诊断链路完整 |

### 记忆系统升级策略（两阶段方案）

当前的 `memory_manager.py` 是纯内存实现，基于 TF-IDF 的简单检索。从这个起点直接跳到 ChromaDB 向量数据库跨度过大，因此拆分为两个阶段：

1. **S9-01a（P0）——持久化 + LLM 摘要归档**：将记忆存储迁移至 SQLite（复用已有的 `storage/db.py`），引入 LLM 摘要归档机制。当记忆条目超过阈值（如 50 条）时，自动调用 LLM 将旧记忆压缩为摘要并归档。这一步不引入任何新的外部依赖。
2. **S9-01b（P1）——向量检索集成**：在 S9-01a 稳定运行后，引入 ChromaDB 做语义检索，替换 TF-IDF。仅在 P0 全部完成后才启动。

### 多模型测试策略

S9-02 引入多模型路由后，测试矩阵的复杂度将显著上升。必须同时交付以下测试基础设施：

1. **模型评估基准测试**：创建一套固定的“金标准”场景（包含 5 个典型推演场景），对每个模型组合运行并评估输出质量（结构合规性、逻辑一致性、字段完整性）。
2. **CI 集成**：在 GitHub Actions 中新增 `model-eval` Job，仅在手动触发或周期性调度时运行（避免每次 PR 都消耗 API 额度），输出各模型的质量分数报告。
3. **降级抦截**：如果低成本模型的评估分数低于阈值，路由器自动回退到高质量模型并记录告警。

### 关键依赖

- S9-01a 依赖已有的 `storage/db.py` 和 `memory_manager.py` 接口。
- S9-01b 依赖 S9-01a 稳定运行。
- S9-03 与 S9-04 依赖 Sprint 7-8 已稳定的状态结构与追踪能力。
- S9-02 依赖 Sprint 7 的统一调用路径。
- S9-06 与 S9-07 是 Sprint 9 上线前的发布护栏。

### Sprint 退出标准

- 长篇场景下关键人物、地点、规则能被召回并继续使用。
- 路由后的成本统计可追踪，且逻辑层成本有明确下降。
- 低成本模型的评估分数不低于阈值，否则自动回退。
- Wiki、富文本、草稿恢复形成完整创作闭环。

## 6. 非承诺范围

以下事项不纳入 Sprint 7-9 的承诺交付，避免吞噬主线产能：

- **服务端**：全域微服务拆分或合并、全域 DDD 深度重构、全量零信任推广。
- **AI 系统**：完整 Git 式版本控制、复杂的自动节奏判断。
- **前端**：独立遥测系统产品化界面，本期只集成在现有侧栏。

| 风险 | 影响 | 应对 |
