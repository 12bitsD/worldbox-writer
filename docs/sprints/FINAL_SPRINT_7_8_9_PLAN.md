## 1. 这份计划解决什么问题

这份计划用于把已批准的最终决策改写成可执行的 Agile Sprint 计划，明确 Sprint 7-9 的目标、范围、团队分工、验收标准与风险边界。

### 核心排期原则

1. **产品目标优先**：不做脱离用户价值的纯技术重构，技术改造只作为功能交付的护栏。
2. **里程碑连续推进**：Sprint 6 建立“看见世界”的最小闭环，Sprint 7 补齐其稳定性与体验，Sprint 8 解决“掌控世界”，Sprint 9 解决“创作作品”。
3. **每个 Sprint 都必须可演示**：每轮结束都要有前后端联调可见的增量，而不是只完成底层准备。

**当前状态**：Delivered on 2026-04-18

Sprint 7 的目标不是从零开始做“看见世界”，而是在 Sprint 6 最小闭环已经建立的前提下，补齐可读性、稳定性和统一调用链。

- 用户能获得**更完整、更易读的关系图谱体验**。
- 用户能获得**可筛选、可关联的 Agent 遥测日志**。
- 系统在**大量节点流式推演**时保持可用，不因前端渲染或接口抖动失稳。

## 4. Sprint 8 计划

### Sprint Goal

Sprint 8 的目标不是一次性做完“所有分支能力”，而是交付第一个真正可用的 Control 闭环：用户能从已存在的历史节点创建分支、继续推演、切换世界线，并做基础对比。

### 业务结果

- 用户能从历史节点创建一条新世界线，并在该分支上继续推演。
- 用户能在前端查看、切换并比较主线与支线的基础结果。
- 系统具备 branch-safe 的存储、追踪和发布护栏，避免新分支能力把既有主线会话打坏。

### Sprint Backlog 组织方式

Sprint 8 不再把“用户功能”“底层使能”“上线护栏”全部写成并列 P0，而是按三层组织：

- **承诺交付的用户故事（P0）**：直接让用户获得“分叉并控制世界线”的能力。
- **支撑这些故事的工程使能项（P0）**：只保留 Sprint 8 成功所必需的底盘。
- **发布护栏与伸缩项（P1）**：对首个分支版本有价值，但不应反客为主拖垮 Sprint Goal。

这样组织的原因很直接：当前代码库只预留了 `branch_id` / `merged_from_ids` / `branches` / `active_branch_id`，并没有真正的分支引擎、历史快照恢复机制或分支 UI。如果继续把“多 Actor 协商”“Zod 全量校验”“自动回滚”等事项并列为 P0，Sprint 8 会再次失去收口点。

### 架构约束：先解决“可恢复”，再谈“可分叉”

当前 `StoryNode` 已预留 `branch_id` 和 `merged_from_ids`，`WorldState` 已预留 `branches` 与 `active_branch_id`，但现有存储层只会把最终 `WorldState` 整体序列化到 `state_json`。这意味着：

- 当前并不能从任意历史节点“还原当时的世界状态”。
- 仅靠 `branch_id` 字段，无法可靠实现真正的 `fork_at_node()`。
- 由于 LLM 推演并非确定性过程，事后重放整条历史链再分叉不是可靠的默认方案。

所以 Sprint 8 的第一优先级不是直接做 Timeline UI，而是先交付 Branch Seed Snapshot v1。
v1 的原则是务实可用：优先保证“能从历史节点恢复并续跑”，而不是在本期追求最优存储压缩。若需要，可先采用完整 `WorldState` 快照或其他明确可恢复的 fork seed 表示，后续再优化为 diff-based 方案。

### 明确不作为本期承诺 P0 的事项

- **多 Actor 协商机制**：它更像未来对推演质量的升级，不是首个 branching loop 的必要前置，移出 Sprint 8 承诺范围。
- **前端全量 Zod 运行时校验**：当前前端并未把 Zod 作为直接依赖，也没有既定的运行时 schema 校验体系。Sprint 8 先保证后端契约、类型同步和错误边界，不强行把 Zod 立成 P0。
- **自动回滚**：当前仓库没有成型的灰度平台或自动回滚编排能力。本期只承诺 feature flag + manual rollback runbook。
- **分支合并（Reconvergence）功能**：继续保留架构预留，但不在本期实现真实 merge。

### 关键依赖

- US-06.01 依赖 S8-EN-01 和 S8-EN-02 先补齐可恢复的 fork seed 与分支契约。
- US-06.02 依赖 US-06.01 已产出稳定的 branch registry、active branch 语义和基础对比字段。
- US-06.03 依赖 Sprint 7 已冻结的调用链与 telemetry 字段，并要求分支链路能附带节奏上下文。
- S8-EN-04 是发布前置条件，但不阻塞本地开发和联调。
- `merged_from_ids` 继续只做架构预留；本期不允许因为“顺手做 merge”而扩大范围。

### Sprint 退出标准

- 用户能在真实界面中，从同一历史节点创建至少 2 条分支，并分别继续推演。
- 切换分支后，REST 初始载荷、历史恢复和页面刷新看到的是同一条活跃世界线，不发生跨分支串写。
- 用户至少能比较 `forked_from_node`、节点数、最后节点摘要、最新 tick 这 4 类基础信息。
- 分支链路中的节点、Telemetry 与 Trace 至少能关联到 `branch_id`。
- 分支能力可通过 feature flag 显式开启/关闭；关闭后系统退回单主线安全行为。
- P1 项未完成时，不阻塞 Sprint 8 作为 v0.7.x 首个 branching release 结束。

## 5. Sprint 9 计划

### Sprint Goal

Sprint 9 的目标是补齐长篇创作所需的记忆、成本与编辑能力，让产品从“可玩”走向“可长期使用”。

### 业务结果

- 长篇推演的一致性明显提升，不再频繁遗忘设定。
- 模型调用成本可控，逻辑推演与文本渲染按场景分流。
- 用户能在 Wiki 与富文本编辑器中继续整理和打磨内容。

### Sprint Backlog

| ID | 条目 | 优先级 | 负责人 | 说明 | 验收结果 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| S9-01a | 记忆持久化 + LLM 摘要归档 | P0 | AI 系统团队 | 将内存记忆迁移至 SQLite，引入 LLM 摘要归档机制，当记忆条目超过阈值时自动压缩旧记忆为摘要 | 旧记忆被压缩后仍可被召回，新记忆重启后不丢失 |
| S9-01b | 向量检索集成 (ChromaDB) | P1 | AI 系统团队 | 在 S9-01a 稳定后，引入 ChromaDB 做语义检索，替换当前的 TF-IDF 简单匹配 | 语义相关的设定能被召回，即使关键词不完全匹配 |

### 记忆系统升级策略（两阶段方案）

当前的 `memory_manager.py` 是纯内存实现，基于 TF-IDF 的简单检索。从这个起点直接跳到 ChromaDB 向量数据库跨度过大，因此拆分为两个阶段：

1. **S9-01a（P0）——持久化 + LLM 摘要归档**：将记忆存储迁移至 SQLite（复用已有的 `storage/db.py`），引入 LLM 摘要归档机制。当记忆条目超过阈值（如 50 条）时，自动调用 LLM 将旧记忆压缩为摘要并归档。这一步不引入任何新的外部依赖。
2. **S9-01b（P1）——向量检索集成**：在 S9-01a 稳定运行后，引入 ChromaDB 做语义检索，替换 TF-IDF。仅在 P0 全部完成后才启动。

### 多模型测试策略

S9-02 引入多模型路由后，测试矩阵的复杂度将显著上升。必须同时交付以下测试基础设施：

1. **模型评估基准测试**：创建一套固定的“金标准”场景（包含 5 个典型推演场景），对每个模型组合运行并评估输出质量（结构合规性、逻辑一致性、字段完整性）。
2. **CI 集成**：在 GitHub Actions 中新增 `model-eval` Job，仅在手动触发或周期性调度时运行（避免每次 PR 都消耗 API 额度），输出各模型的质量分数报告。
3. **降级抦截**：如果低成本模型的评估分数低于阈值，路由器自动回退到高质量模型并记录告警。

### 关键依赖

- S9-01a 依赖已有的 `storage/db.py` 和 `memory_manager.py` 接口。
- S9-01b 依赖 S9-01a 稳定运行。
- S9-03 与 S9-04 依赖 Sprint 7-8 已稳定的状态结构与追踪能力。
- S9-02 依赖 Sprint 7 的统一调用路径。
- S9-06 与 S9-07 是 Sprint 9 上线前的发布护栏。

### Sprint 退出标准

- 长篇场景下关键人物、地点、规则能被召回并继续使用。
- 路由后的成本统计可追踪，且逻辑层成本有明确下降。
- 低成本模型的评估分数不低于阈值，否则自动回退。
- Wiki、富文本、草稿恢复形成完整创作闭环。
