## 1. Sprint 19 要解决什么问题

等待干预态原本把“关键节点”和“编辑设定”作为两个大面板堆在故事流底部。用户在阅读推演结果时会被大块控件打断，尤其在关键节点弹出后，界面重心从故事正文偏移到控制面板。

Sprint 19 的目标是把干预能力从“占据主阅读区的大面板”改成“轻量提示 + 按需展开抽屉”，同时复查故事流和三栏布局，把默认视线还给小说正文。

---

## 2. Sprint Goal

**让关键节点不挡阅读，默认展示故事正文，把干预、设定编辑和调试信息收敛到按需打开的工作区。**

Sprint 19 结束时，项目具备：

- 紧凑的等待干预条
- 右侧抽屉式干预工作区
- 设定编辑嵌入同一抽屉
- StoryFeed 默认优先显示小说正文
- 逻辑摘要与 SceneScript 折叠展示
- 右侧推演信息面板可折叠
- 主布局移动端降级为单列
- 前端组件回归测试

---

## 3. 方案骨架

### 3.1 承诺交付

1. InterventionPanel 从大块面板改为 sticky compact bar + drawer
2. EditPanel 支持 embedded 模式并进入干预抽屉
3. StoryFeed 以 rendered text / streaming text 为默认阅读主体
4. SceneScript 与逻辑摘要默认折叠，保留可追踪性
5. 右侧推演信息支持手动收起
6. 主布局增加响应式降级样式
7. 前端组件测试覆盖新的干预交互和阅读优先行为

### 3.2 非目标

- 不修改后端 API 契约
- 不改变关键节点判定逻辑
- 不改变 Actor / Critic / GM / Narrator 行为
- 不新增复杂的全局布局状态持久化

---

## 4. 方案收敛记录

### Round 1：继续保留底部大面板，只缩小高度

问题：

- 仍然挤压故事流底部空间
- 编辑设定面板仍会和干预面板叠加
- 小屏下仍然容易挡住阅读

结论：否决。

### Round 2：改成弹窗 Modal

问题：

- Modal 会遮住背后的故事推演，不符合“边看边干预”
- 干预上下文和故事正文之间的空间关系被切断

结论：否决。

### Round 3：轻量条 + 侧边抽屉

采用方案：

- 默认只保留一条关键节点提示，不抢主阅读区
- 用户主动点击后，右侧打开抽屉进行干预或设定编辑
- 抽屉不覆盖中心故事正文的主体阅读区域
- 逻辑信息默认折叠，读者先看正文，创作者需要时再展开证据

结论：采用。

- 等待干预时，故事主阅读区不再被大块干预控件占据

## 7. 为什么这个范围满足设计预期

Sprint 19 没有继续增加 Agent 能力，而是修复“读者视线被控制面板夺走”的体验问题。WorldBox Writer 的核心交互是先看世界推演，再在关键点干预；因此默认状态应该保护故事阅读，只有在用户明确需要时才展开控制工作区。
