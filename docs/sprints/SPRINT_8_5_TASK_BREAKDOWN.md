# Sprint 8.5 开发任务拆解

**文档状态**：Executed  
**适用 Sprint**：Sprint 8.5  
**最后更新**：2026-04-19

本文档用于把 Sprint 8.5 的“首个可见反馈与渐进出字优化”继续拆到开发任务层，供实现、PR 切分与验收使用。

---

## 1. 承诺范围

### P0 用户故事

- `US-06.04` 首次推演的阶段进度可见化与首幕渐进反馈

### P0 工程使能项

- `S8.5-EN-01` 初始化阶段 Telemetry 可见化
- `S8.5-EN-02` 首个正文关键路径裁剪
- `S8.5-EN-03` Progressive Feedback UI

### P1 伸缩项

- `S8.5-EN-04` SSE 首包与缓冲优化
- `S8.5-EN-05` 角色级模型路由配置钩子

---

## 2. 推荐开发顺序

建议按以下顺序推进：

1. `S8.5-EN-01`
2. `S8.5-EN-04`
3. `S8.5-EN-03`
4. `S8.5-EN-02`
5. `S8.5-EN-05`

原因：

- 先把初始化阶段“看不见进度”的问题修掉，马上就能改善体验。
- 再把 SSE 首包和前端连接顺序理顺，避免前端自己人为延迟首条事件。
- Progressive UI 可以尽快消费这些事件，形成真实用户感知。
- 最后再改引擎关键路径，属于体验优化里风险稍高但收益最大的部分。

## 3. 需求拆解

## 3.1 S8.5-EN-01 初始化阶段 Telemetry 可见化

**目标**

- 在 `world` 尚未完成构建时，也能把初始化阶段进度返回给前端。

**期望收口**

- `GET /simulate/{id}` 在 `world is None` 时仍返回 telemetry。
- 前端启动后不需要等首个完整世界快照才能开始展示“系统正在做什么”。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S8.5-EN-01-T1 | 调整 API payload builder，保留 initializing 阶段 telemetry | 服务端架构团队 | 无 | 后端实现 |
| S8.5-EN-01-T2 | 启动新会话时先挂前端占位 state，再立即订阅 SSE | 前端团队 | T1 | Hook 改造 |
| S8.5-EN-01-T3 | 增加 API 测试，覆盖“无 world 但有 telemetry”的场景 | 服务端架构团队 | T1 | pytest |

**验收检查**

- 初始化阶段至少一条 telemetry 能经由 REST 或 SSE 被前端消费。
- 用户不再只能看到固定 loading 文案。

---

## 3.2 S8.5-EN-04 SSE 首包与缓冲优化

**目标**

- 降低首条流式事件因为队列轮询和缓冲策略带来的额外等待。

**期望收口**

- 事件流从“固定 sleep 轮询”改为“有事件即推”。
- SSE 响应头更明确地表达 no-cache / no-buffer 意图。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S8.5-EN-04-T1 | 调整服务端 SSE generator，减少 500ms 轮询空窗 | 服务端架构团队 | 无 | 后端实现 |
| S8.5-EN-04-T2 | 评估并补齐反向代理友好的 SSE headers | 服务端架构团队 | T1 | 响应头优化 |
| S8.5-EN-04-T3 | 通过现有 API 测试验证事件流未被破坏 | 服务端架构团队 | T2 | 回归验证 |

**验收检查**

- 首条 `status` / `telemetry` / `narrator_start` 事件不再被额外固定 sleep 卡住。

---

## 3.3 S8.5-EN-03 Progressive Feedback UI

**目标**

- 把已有 telemetry 从“诊断面板信息”升级成“主区域进度反馈”。

**期望收口**

- 用户在主区域能看到当前阶段、最近进度和是否进入正文流式状态。
- 右侧 Telemetry 面板仍保留诊断属性，但不再是唯一反馈入口。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S8.5-EN-03-T1 | 定义阶段分组：初始化世界、生成第一幕、渐进出字、补全世界细节 | 前端团队 | S8.5-EN-01 | 阶段映射 |
| S8.5-EN-03-T2 | 新增主区域 Progress Panel，消费 telemetry 生成阶段状态 | 前端团队 | T1 | React 组件 |
| S8.5-EN-03-T3 | 在 `App` 中接入 Progress Panel，并优化无世界 / 无节点时的空状态 | 前端团队 | T2 | UI 集成 |
| S8.5-EN-03-T4 | 增加前端测试，覆盖进度映射与渲染 | 前端团队 | T2, T3 | Vitest |

**验收检查**

- 没有节点时，主区域也能展示正在执行的阶段。
- 开始 narrator 流式时，UI 会显式切换为“正在出字”。

## 3.4 S8.5-EN-02 首个正文关键路径裁剪

**目标**

- 让 `WorldBuilder` 不再阻塞第一幕正文开始流式输出。

**期望收口**

- 第一轮故事生成路径变为：`Director -> Actor -> GateKeeper -> NodeDetector -> Narrator`。
- `WorldBuilder` 仍保留，但延后到第一幕可见之后执行。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S8.5-EN-02-T1 | 调整 LangGraph 路由，让 `Director` 后直接进入 `Actor` | AI 系统团队 | 无 | 图路由修改 |
| S8.5-EN-02-T2 | 定义 `Narrator` 之后何时补跑 `WorldBuilder`，避免破坏 waiting / complete 语义 | AI 系统团队 | T1 | 路由规则 |
| S8.5-EN-02-T3 | 增加引擎单元测试，覆盖 deferred world builder 的分支逻辑 | AI 系统团队 | T2 | pytest |
| S8.5-EN-02-T4 | 检查前端世界面板对“世界细节稍后补全”是否有可接受空状态 | 前端团队 | T2 | UX 确认 |

**验收检查**

- 第一幕正文开始流式输出前，不再依赖完整 `world_builder` 完成。
- 之后的世界详情仍能补齐，不破坏完整会话结果。

## 3.5 S8.5-EN-05 角色级模型路由配置钩子

**目标**

- 为后续 latency tuning 预留更细的模型路由入口，但不把 provider 选择逻辑写死在本期实现里。

**期望收口**

- 可以通过环境变量按角色覆盖模型，但默认行为保持不变。

**开发任务**

| Task ID | 开发任务 | Owner | 依赖 | 产物 |
| :--- | :--- | :--- | :--- | :--- |
| S8.5-EN-05-T1 | 设计 `LLM_MODEL_<ROLE>` 级覆盖规则 | AI 系统团队 | 无 | 配置约定 |
| S8.5-EN-05-T2 | 在 `utils/llm.py` 中补优先级解析 | AI 系统团队 | T1 | 实现 |
| S8.5-EN-05-T3 | 更新 README / Runbook，说明如何单独调快 `director/world_builder` | 服务端架构团队 | T2 | 文档更新 |

**验收检查**

- 不改代码即可对不同角色做模型实验。
- 未配置时行为与当前完全一致。

## 5. 验收问题清单

开发完成后至少回答以下问题：

1. 启动新推演后，首个可见反馈是否比之前更早？
2. 主区域是否已经不再依赖完整世界快照才能显示“系统在工作”？
3. 第一幕正文开始流式输出前，`WorldBuilder` 是否已从关键路径移走？
4. 若 `WorldBuilder` 延后执行，是否破坏了 `waiting` / `complete` / 分支语义？
5. 默认门禁是否仍通过：`make lint`、`make test`？
